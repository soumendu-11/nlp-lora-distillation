# Cancer Abstract Classification вҖ” LoRA vs Knowledge Distillation

Comparative study of two parameter-efficient / teacher-guided training strategies for binary biomedical text classification (cancer vs non-cancer abstracts).

---

## Overview

| Approach | Description |
|---|---|
| **LoRA** | DistilBERT fine-tuned with Low-Rank Adaptation (r=4, Оұ=32) вҖ” only ~0.93 % of parameters are trained |
| **Distillation** | DistilBERT student trained on soft probability labels generated by Azure GPT-4o вҖ” combines KL-divergence and cross-entropy loss |

Both models achieve **96 % test accuracy / 95.83 % F1** on 150 held-out biomedical abstracts.

---

## Dataset

1 000 biomedical PubMed abstracts (500 cancer, 500 non-cancer) stored as `.txt` files under `Dataset/`.

| Split | Samples |
|---|---|
| Train | 700 (70 %) |
| Validation | 150 (15 %) |
| Test | 150 (15 %) |

### Why are the metrics identical?

The models differ on exactly **2 out of 150 test predictions** вҖ” in a perfectly symmetric swap вҖ” so aggregate metrics cancel out. See Section 11 of `results.ipynb` for a detailed breakdown.

**Key factors:**
- 80 % of GPT-4o soft labels were near-binary (P > 0.9 or P < 0.1), giving distillation no extra signal beyond hard labels
- LoRA reached 96 % validation accuracy in epoch 1 вҖ” the task is simple enough for both approaches to find the same decision boundary quickly
- 150 test samples is too small to detect the 2-prediction difference in aggregate metrics

## LoRA Architecture

- **Base model**: `distilbert-base-uncased`
- **LoRA rank (r)**: 4
- **LoRA alpha**: 32
- **Target modules**: `q_lin` (query projection in each attention layer)
- **Trainable params**: 628,994 / 67.6 M (0.93 %)

## Distillation Setup

- **Teacher**: Azure GPT-4o (zero-shot soft probability generation)
- **Student**: `distilbert-base-uncased` (all parameters trained)
- **Temperature (T)**: 4
- **Alpha (Оұ)**: 0.7 (KL weight); 0.3 (cross-entropy weight)
- **Loss**: `L = ОұВ·TВІВ·KL(student/T вҖ– teacher) + (1вҲ’Оұ)В·CE(student, hard_label)`
- **Soft label cache**: `data_splits/soft_labels_cache.json` (avoids repeated API calls)

### Hyperparameter Explanations

#### Why Temperature (T) = 4?

Temperature controls how much the student's probability distribution is **softened** before computing the KL-divergence loss. The formula is `softmax(logits / T)` вҖ” dividing by T shrinks the logits before softmax, which compresses the gap between class probabilities and produces a more spread-out distribution.

**Effect of T on a sample logit `[3.0, 0.5]`:**

| Temperature | Softmax output | Effect |
|---|---|---|
| T = 1 | [0.92, 0.08] | Sharp вҖ” nearly certain |
| T = 2 | [0.81, 0.19] | Softer |
| **T = 4** | **[0.68, 0.32]** | **Balanced spread (chosen)** |
| T = 10 | [0.56, 0.44] | Nearly uniform |

**Why T = 4 specifically?** It is the standard sweet spot from Hinton et al.'s original 2015 distillation paper:

- **Too low (T = 1вҖ“2)** вҖ” the student barely learns from soft labels; the distribution stays sharp and near-binary, effectively reducing the loss to plain cross-entropy on hard labels.
- **Too high (T = 8вҖ“10+)** вҖ” differences between classes get washed out completely; probabilities look nearly uniform and the training signal degrades.
- **T = 4** вҖ” softens the output just enough to amplify the "dark knowledge" (the small but meaningful probabilities the teacher assigns to wrong classes) without losing the overall signal.

> **Note on this project:** Because GPT-4o emitted near-binary labels (P > 0.9 for one class in ~80% of samples), the temperature scaling had limited practical effect вҖ” no amount of softening on the student side can recover nuance the teacher never expressed.

---

#### What is Alpha (Оұ) and Why Оұ = 0.7?

Alpha controls the **balance between two loss terms**: learning from the teacher (soft labels) vs. learning from ground truth (hard labels).

**The combined loss with Оұ = 0.7:**

```
L = 0.7 В· TВІ В· KL(student/T вҖ– teacher) + 0.3 В· CE(student, hard_label)
        в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ soft-label loss в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җ hard-label loss в”Җв”Җв”ҳ
```

**What each term does:**

| Term | Learns from | Signal |
|---|---|---|
| **KL divergence** (weighted by Оұ) | Teacher's soft probabilities (GPT-4o) | "Mimic the teacher's uncertainty" |
| **Cross-Entropy** (weighted by 1 вҲ’ Оұ) | Hard ground-truth labels (0 or 1) | "Predict the correct class" |

**Intuition for choosing Оұ:**

- Оұ вҶ’ **1.0** вҖ” trust the teacher entirely, almost ignore ground truth
- Оұ вҶ’ **0.0** вҖ” ignore the teacher, fall back to standard fine-tuning
- **Оұ = 0.7** вҖ” lean toward the teacher (70%) but keep ground truth as a safety anchor (30%) in case the teacher is wrong

**Why is the KL term multiplied by TВІ?**
This is a mathematical correction from Hinton's paper. Dividing logits by T shrinks the KL gradients by a factor of 1/TВІ. Multiplying the loss by TВІ restores the gradients to the same scale as the CE loss, so neither term dominates simply because of temperature scaling.

### Libraries & Implementation Notes

There is **no dedicated distillation library** in this project вҖ” the distillation logic is custom-implemented from scratch using PyTorch primitives.

**Core libraries used in `src/distillation_training.py`:**

| Library | Purpose |
|---|---|
| **PyTorch** (`torch`) | Training loop, tensors, optimizer, KL-divergence and cross-entropy losses |
| **HuggingFace Transformers** | Loading `distilbert-base-uncased` student model + tokenizer |
| **Azure OpenAI SDK** (`openai.AzureOpenAI`) | Calling GPT-4o teacher to generate soft labels |
| **scikit-learn** | Computing accuracy, F1, precision, recall metrics |
| **python-dotenv** | Loading Azure API credentials from `.env` |
| **tqdm** | Progress bars |

The distillation loss is hand-written in `src/distillation_training.py` (lines 156вҖ“173) using just two PyTorch functions:

```python
torch.nn.functional.kl_div(...)         # KL divergence term
torch.nn.functional.cross_entropy(...)  # hard-label term
```

**Why no specialized distillation library?**

For binary classification with logit-based KL distillation, raw PyTorch is sufficient and more transparent than wrapping higher-level frameworks like:

- `textbrewer` (general-purpose distillation toolkit)
- `huggingface/distil*` reference implementations
- `torchdistill`

Those would only be needed for multi-stage distillation, intermediate-layer matching (attention/hidden-state transfer), or a config-driven pipeline. A direct PyTorch implementation keeps the loss math fully visible and easy to tune.

> **Note:** LoRA (the other approach in this project) *does* use a dedicated library вҖ” **PEFT (`peft>=0.12.0`)** from HuggingFace вҖ” for the low-rank adapter injection. Distillation here is plain PyTorch.
